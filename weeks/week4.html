1. Operating-System Services

What Are OS Services?

An operating system provides an environment for execution of programs and services to programs and users. These services differ from one OS to another, but certain common classes of services exist across all operating systems.

OS services can be divided into two categories: those that are helpful to the user and those that ensure efficient system operation.

Services Helpful to the User

User Interface (UI)

Nearly all operating systems provide a user interface. This can take several forms:

  • Command-Line Interface (CLI) - Uses text commands entered via keyboard
  • Graphics User Interface (GUI) - Window-based system with pointing device (mouse, trackpad)
  • Touch-Screen Interface - Gestures on a touch display (press, swipe, pinch)
  • Batch Interface - Commands and directives entered into files; executed without direct user interaction

Program Execution

The system must be able to load a program into memory and run it. The program must be able to end its execution, either normally or abnormally (indicating an error).

I/O Operations

A running program may require I/O, involving a file or an I/O device. Users usually cannot control I/O devices directly, so the operating system must provide a means to do I/O.

File-System Manipulation

Programs need to read and write files and directories, create and delete them, search for them, list file information, and manage permissions.

Communications

Processes may need to exchange information with each other, on the same computer or between networked computers. Communication may be implemented via two models:

  • Shared Memory - Two or more processes read and write to a shared region of memory
  • Message Passing - Packets of information are moved between processes by the operating system

Error Detection

The OS needs to be constantly aware of possible errors. Errors may occur in:

  • CPU and memory hardware (e.g., memory error, power failure)
  • I/O devices (e.g., parity error on disk, connection failure on network)
  • User programs (e.g., arithmetic overflow, attempt to access illegal memory location)

For each type of error, the OS should take the appropriate action to ensure correct and consistent computing.

Services for Efficient System Operation

Resource Allocation

When multiple processes are running concurrently, resources must be allocated to each of them. The OS manages many types of resources:

  • CPU scheduling - Determines which process gets the CPU and for how long
  • Memory allocation - Assigns memory blocks to processes
  • Device allocation - Manages access to I/O devices

Logging

The OS tracks how much and what kinds of computer resources each user or process uses. This record keeping may be used for:

  • Accounting - Usage statistics for billing purposes
  • Performance tuning - Accumulating usage data to reconfigure the system for improved performance

Protection and Security

When several processes execute concurrently, it should not be possible for one process to interfere with the others or with the operating system itself:

  • Protection - Ensures that all access to system resources is controlled
  • Security - Defends against external threats (e.g., denial-of-service attacks, worms, viruses) and requires user authentication to gain access to system resources

Key Insight: OS services form the bridge between user needs and hardware capabilities. Every action a user program performs -- from reading a file to displaying output -- ultimately relies on an OS service.

2. A View of Operating-System Services

Layered View of OS Services

Operating-system services can be visualized as a layered architecture where each layer provides services to the layer above and uses services from the layer below. This view shows how user programs interact with the OS through well-defined interfaces.

Overview of Operating-System Services

Users user 1   |   user 2   |   user 3   |   ...   |   user n
System and Application Programs compiler   |   assembler   |   text editor   |   database system
User Interfaces GUI   |   batch   |   command line
System Calls
OS Services program execution   |   I/O operations   |   file systems   |   communications   |   error detection
OS Kernel Services resource allocation   |   logging   |   protection & security
Hardware CPU   |   memory   |   disks   |   network   |   I/O devices

Layered view showing how user programs access hardware through system calls and OS services. User-facing services sit above the system-call interface, while resource management services operate below.

Key Insight: The system-call interface acts as a boundary between user-level programs and kernel-level services. Everything above the system-call line runs in user mode; everything below runs in kernel mode.

3. Command-Line Interpreter (CLI)

What is a CLI?

The Command-Line Interpreter (also called a shell) allows direct command entry from the user. It is sometimes implemented in the kernel, sometimes by system programs. On systems with multiple command interpreters, the interpreters are known as shells.

The CLI fetches a command from the user and executes it. The primary function is to get and execute the next user-specified command.

How CLI Commands Work

Commands can be implemented in two ways:

Built-in Commands

The command interpreter itself contains the code to execute the command. The number of commands determines the size of the CLI.

System Programs

Commands are implemented through system programs. The CLI does not understand the command; it simply uses the command to identify a file to be loaded and executed. For example:

$ rm file.txt

The shell searches for a file called rm, loads it into memory, and executes it with the parameter file.txt.

Popular Shells

Common Unix/Linux Shells

Shell Name Description
sh Bourne Shell The original Unix shell, written by Stephen Bourne. Simple and portable.
bash Bourne Again Shell Enhanced version of sh; the default on most Linux distributions.
csh / tcsh C Shell C-like syntax; popular among BSD Unix users.
ksh Korn Shell Combines features of sh and csh; used in commercial Unix systems.
zsh Z Shell Extended Bourne shell with many improvements; default on macOS.

Bourne Shell Command Example

Bourne Shell (sh)
$ ls -la /home total 12 drwxr-xr-x 3 root root 4096 Jan 10 09:00 . drwxr-xr-x 20 root root 4096 Jan 10 09:00 .. drwxr-xr-x 5 user user 4096 Jan 10 10:30 user $ cp source.txt dest.txt $ echo "Hello, OS World!" Hello, OS World!

Key Insight: The shell is just a user program -- it is not part of the kernel. Adding new commands does not require modifying the shell when commands are implemented as external programs.

4. Graphical User Interface (GUI)

What is a GUI?

A Graphical User Interface (GUI) provides a user-friendly desktop metaphor interface. Instead of typing commands, users interact with visual elements using a pointing device (mouse, trackpad) and keyboard.

Desktop Metaphor

The GUI uses a desktop metaphor where the screen represents a desk, and files are represented as icons on that desk:

  • Icons represent files, programs, folders, and system actions
  • Mouse/pointer is used to navigate, select, and interact with objects
  • Windows display the contents of folders and running applications
  • Menus provide lists of available actions and settings

Touch-Screen Interfaces

Modern devices have moved beyond the traditional mouse-based GUI to touch-screen interfaces:

  • Gestures replace mouse clicks -- tap, swipe, pinch-to-zoom
  • Virtual keyboard appears on screen when text input is needed
  • No mouse or physical keyboard needed for basic interaction
  • Common on smartphones and tablets (iOS, Android)

Mac OS X Interface

Mac OS X provides a well-known example of a modern GUI, featuring:

  • Aqua interface with the Dock for launching applications
  • Finder for file system navigation
  • Spotlight for system-wide search
  • Combines GUI with a powerful Unix-based CLI (Terminal.app)

Key Insight: Many systems now include both CLI and GUI interfaces. System administrators and power users often prefer the CLI for efficiency and scriptability, while casual users prefer the GUI for ease of use. Most operating systems provide both.

5. System Calls

What Are System Calls?

System calls provide the programming interface to the services provided by the operating system. They are the primary mechanism by which a user-level program requests a service from the kernel. System calls are typically written in a high-level language (C or C++), although some low-level tasks may require assembly language.

Application Programming Interfaces (APIs)

Rather than invoking system calls directly, programs mostly access them through a high-level Application Programming Interface (API). The three most common APIs are:

Major System Call APIs

Win32 API

For Windows systems

Used by all Windows applications; provides access to Windows kernel services.

POSIX API

For POSIX-based systems

Covers virtually all versions of UNIX, Linux, and macOS. Defined by IEEE standard.

Java API

For the Java Virtual Machine

Platform-independent API; JVM translates to native system calls.

Example: Copying a File

A simple operation like copying a file involves many system calls. Consider the sequence of operations:

System Calls in a File Copy Operation

1
Acquire input file name

Prompt user / read from command line

2
Open input file

System call: open() -- may fail if file does not exist

3
Acquire output file name

Prompt user / read from command line

4
Create output file

System call: create() -- may fail if file already exists

5
Loop: Read from input, Write to output

System calls: read() and write() -- repeated until end of file

6
Close both files

System calls: close() for both input and output files

7
Terminate program

Display completion message or error; return to shell

Even a simple file copy requires dozens of system calls behind the scenes.

Standard API Example: read()

POSIX read() System Call

ssize_t read(int fd, void *buf, size_t count)
Parameter Type Description
fd int File descriptor -- an integer identifying the open file
buf void * Pointer to a buffer where data will be stored
count size_t Maximum number of bytes to read
Returns: Number of bytes actually read, 0 at end of file, or -1 on error.

Key Insight: Programmers rarely use system calls directly. Instead, they use higher-level APIs that internally translate into the appropriate system calls. This abstraction makes programs more portable across different operating systems.

6. System Call Implementation

How System Calls Work

Each system call is associated with a number. The system-call interface maintains a table indexed by these numbers. When a system call is invoked, the interface invokes the intended system call in the OS kernel and returns the status and any return values.

The System Call Mechanism

The caller does not need to know anything about how the system call is implemented -- it only needs to know the API and understand what the OS will do as a result of the call:

  • The API hides most details of the OS interface from the programmer
  • The runtime support library (e.g., C library) manages the system-call interface
  • Parameters are passed via registers, stack, or memory blocks depending on the OS

Parameter Passing Methods

Methods for Passing Parameters to System Calls

Method How It Works Limitations
Registers Parameters passed in CPU registers Limited by number of available registers
Block / Table Parameters stored in a memory block; address of block passed in a register Extra memory access needed
Stack Parameters pushed onto the stack by the program; popped off by the OS Stack size limitations

The block and stack methods do not limit the number or length of parameters being passed. Linux uses a combination of registers and stack.

Key Insight: The system-call interface provides a clean separation between user programs and the kernel. The user program sees only the API; the kernel implements the actual service. This separation allows either side to change without affecting the other.

Summary and Key Takeaways

What We've Learned

  1. OS Services provide essential functionality - The operating system offers services for user interfaces, program execution, I/O, file systems, communications, and error detection to support user programs.
  2. System operation services ensure efficiency - Resource allocation, logging, and protection/security services ensure the system runs efficiently and securely for all users.
  3. User interfaces come in many forms - CLI, GUI, touch-screen, and batch interfaces each serve different use cases. Modern systems typically offer both CLI and GUI.
  4. The CLI (shell) is a powerful tool - Shells like Bourne Shell, bash, and zsh allow direct command execution. Commands can be built into the shell or implemented as external programs.
  5. GUIs provide visual interaction - The desktop metaphor with windows, icons, and menus makes computing accessible. Touch interfaces extend this to mobile devices.
  6. System calls are the gateway to OS services - Every OS service is accessed through system calls. Even simple operations like file copy require many system calls.
  7. APIs abstract system calls - Win32, POSIX, and Java APIs provide portable, high-level interfaces so programmers rarely need to make system calls directly.
  8. The system-call mechanism uses a dispatch table - A numbered table maps API calls to kernel functions, with parameters passed via registers, memory blocks, or the stack.
